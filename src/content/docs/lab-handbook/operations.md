# 日常運作

研究室日常運作的目的不是增加行政負擔，而是讓每個人的研究可以被接續、被檢查、被修正。你留下的資料夾、週報、圖、程式、推導、簡報與會議紀錄，是未來自己、老師、學長姐與 AI Agent 都會用到的工作記憶。

## 舊手冊對應

本頁主要改寫自舊手冊第 1 章：老師給的備忘、新生暑期工作、假期與開學前準備、離校前須知、檔案與程式管理、研究生權利義務、Group Seminar、Presentation 與論文倫理。對應 OCR 頁為 005-020。公開版保留操作邏輯，但不公開舊手冊私人資訊與過時工具清單。

## 新生暑期工作

新生暑期工作不是先急著做大題目，而是先把研究室的基本動作練起來。舊手冊要求新生先讀手冊、讀研究與寫作方法、蒐集研究方向資料、學會工具，之後才開始讀書報告。林版改成三個目標：

1. 先建立資料與程式的基本秩序。
2. 走完一個小型地下水資料分析流程。
3. 練習把模型結果翻譯成有邊界的工程判斷。

暑期結束前，至少要留下：

- 一份整理好的資料表。
- 一份可重跑的 Python 或 notebook。
- 一張主圖。
- 一份 5 頁以內的小報告。
- 一段 5 分鐘口頭報告。
- 一份自評：哪些步驟是自己懂的，哪些只是照著做。

## 每週師生會面

會面不是「報告我很努力」，而是把研究推進到下一個可檢查狀態。舊手冊對每週師生會面的要求很具體：會面前要先整理本週進度；把研究內容整理、分析與圖表帶來；如果遇到困難，要先寫下問題與可能解決途徑；會面時要帶齊文獻、計算手稿、輸出結果與列印資料，讓討論可以直接落在證據上。

林版每週交 `weekly-note.md`，格式固定如下：

```text
本週完成：
本週卡住：
我已經試過：
目前證據：
下週要交：
需要老師確認：
```

每次會面前，至少準備三樣東西：

1. 一張圖或一個表，能代表本週最重要的結果。
2. 一段文字，說明這個結果目前支持什麼、不支持什麼。
3. 一個下一步決策：要補資料、改模型、重跑程式、查文獻，還是暫停這條路線。

週報不要寫成心情日記，也不要只寫「繼續看文獻」。如果看文獻，要列出你讀了哪篇、它解決什麼問題、你準備如何用到自己的題目。若臨時無法會面，要提前告知並補交可檢查的進度，不要讓工作週消失。

## 假期與開學前準備

舊手冊要求寒暑假與開學前都要做整理，不是放到最後才補。林版每學期至少做一次 `semester-review.md`：

- 本學期完成哪些資料整理、模型推導、程式、圖與文字。
- 文獻清單是否更新，哪些文獻真的讀過，哪些只是收藏。
- 目前研究問題是否需要收斂、轉向或補實驗。
- 進度表是否合理，下一學期要交什麼可被檢查的產出。
- 程式是否有 readme、輸入輸出說明、變數定義與可重跑案例。
- 雲端、GitHub 與本機是否都有備份，且不是只備份一堆看不懂的舊檔。

假期前要確認資料有備份、儀器或共用電腦狀態已交代、正在跑的工作有人知道風險。長時間離開研究室時，也要讓老師知道你何時回來、哪些事情會延後。

## 檔案與程式管理

舊手冊對程式管理的精神很清楚：變數名稱要統一，輸入輸出格式要一致，重要變數要定義，輸出要能回頭檢查輸入，程式建立後要寫 readme。這些要求在 AI 時代更重要，因為 agent 只能依照你留下的結構判斷。

每個專案至少分成六層：

```text
project/
  README.md
  data_raw/
  data_clean/
  scripts/
  figures/
  manuscript/
  notes/
```

基本規則：

- `data_raw/` 不修改，只保留原始檔。
- `data_clean/` 必須能由程式重建。
- `scripts/` 裡每個程式要有用途說明、輸入檔、輸出檔、主要參數與執行方式。
- `figures/` 的圖要能追到產生它的程式與資料。
- `manuscript/` 放文章草稿、回覆審查與投稿版本。
- `notes/` 放研究筆記、會議紀錄與 AI Agent 摘要。

### 程式 readme

每個可重跑分析至少要有一份程式 readme：

```text
Purpose:
Input files:
Output files:
Main parameters:
How to run:
Expected figures or tables:
Known limitations:
```

如果程式從文字檔讀入參數，輸出檔一開始要 echo print input，也就是把這次使用的輸入條件、參數值與資料檔名列出來。未來你、老師或其他學生看到輸出結果時，才知道這張圖或這組參數到底從哪裡來。

### 四層命名

舊手冊對期刊、研討會與稿件命名很重視，核心不是格式漂亮，而是檔名本身要能判斷內容。林版採四層命名：

```text
YYYYMMDD_topic_action_version.ext
```

四層命名的意思是：

1. 日期：讓版本排序不靠記憶。
2. 主題：讓人知道它屬於哪個研究問題。
3. 動作：例如 clean、fit、plot、draft、reply。
4. 版本：例如 v01、v02、R1、submitted，不要用 `final_final_v3_new.xlsx`。

給老師或共同作者修改的稿件，請維持版本清楚；投稿後的 revision 也要把 R1、R2 與 response letter 對齊。

### 備份

重要資料至少兩份備份：一份在自己的工作機或外接硬碟，一份在雲端或 GitHub。原始資料不壓縮成無法檢查的單一黑箱檔；若必須壓縮，也要保留 manifest，列出裡面有什麼、從哪裡來、何時建立。

## Group Seminar

Group Seminar 不是表演，而是把研究推進到下一步的工作會議。舊手冊把主持人、報告人與安排人的責任分開，這點要保留。

主持人要做：

- 報告前確認投影、時間與順序。
- 控制時間，接近結束前提醒報告人。
- 記錄主要問題、口頭禪與可改進處。
- 討論太分散或太激烈時，把問題拉回研究本身。

報告人需要準備：

1. 研究問題：這週到底要解決什麼？
2. 資料來源：資料從哪裡來，是否已清理？
3. 方法：用了哪個解析模型、數值模型或統計方法？
4. 圖與結果：至少一張可以討論的主圖。
5. 限制：目前哪裡還不能下結論？
6. 下週工作：要做什麼才能排除不確定性？

如果報告的是別人的論文，至少要說清楚 real-world problem、作者處理的問題、使用的方法、主要結果、貢獻、限制，以及這篇文獻能不能改進或吸收到自己的研究。安排人要提前確認報告順序與設備，不要當天才處理。

## Presentation

每次報告都要先想聽眾。專題生、研究生、工程顧問、國際學者與期刊 reviewer 需要聽到的重點不同。舊手冊的簡報原則仍然有效：每張投影片只處理一個重點，多用圖，字要大，對比要清楚，每張不超過八行，報告前要預演。

基本架構：

1. 背景：為什麼這個問題重要？
2. 缺口：傳統方法哪裡不夠？
3. 方法：你的模型或流程如何處理這個缺口？
4. 結果：圖表要回答問題，不要只是展示跑出來的東西。
5. 邊界：資料、模型與應用限制。
6. 下一步：需要哪些資料或實驗才能讓結論更強？

報告時不要照稿念，不要用雷射筆在畫面上亂繞，不要把自己沒把握的推論講成定論。被指出錯誤時，先確認對方意思，記下來，回去檢查。不能當場回答的問題，可以請對方留下聯絡方式，之後用資料或補算結果回覆。

## 學術倫理

舊手冊有一句核心原則：學術研究最重要的是真實。林版延伸為四條：

- 數據不可修改、挑選或美化成想要的樣子。
- 不可一稿兩投，不可把同一個貢獻拆成沒有新意的重複稿。
- 署名要能說明實際貢獻；沒有參與資料、方法、寫作或計畫核心工作的人，不應只因身分而署名。
- 使用他人的圖、表、資料、概念與文字，要明確引用；AI 產出的文字也要回到資料與文獻檢查，不可讓它製造不存在的 claim。

## 離開或轉題前交接

任何人離開研究室、轉題、畢業或暫停專案前，要交接：

- 最新資料夾位置。
- 可重跑的程式與環境說明。
- 目前最好的一張圖。
- 目前最可靠的文字摘要。
- 參考文獻清單與 PDF/筆記位置。
- 重要推導、計算手稿或 notebook。
- seminar、口試或研討會簡報。
- 投稿版本、修改版本與審查回覆。
- 未完成問題清單。
- 不應該繼續使用的錯誤版本。

研究室最怕的不是題目失敗，而是沒有人知道資料在哪裡、哪些結果可靠、哪些結論已被推翻。
